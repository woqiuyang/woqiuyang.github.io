支付模块

支付宝处理交易

先发送同步回调，执行完账户后再异步回调，所有从用户的角度来说异步先到,异步回调需要内网穿透

因为网络波动没有收到回调消息怎么半

自己查

怎么和前端联调

开发阶段的话，ip和前端的是一样的，把ip和端口给他,如果不在一起开发，那么就需要使用内网穿透了

支付宝需要哪些参数

appid 开发者公钥 私钥 支付网关地址 异步回调url

支付会锁库存吗

支付成功才锁库存

支付日志表

记录转账状态和对账数据

用户之前用支付宝支付，如何防止用户再用微信支付

在支付之前判断订单状态
如果已支付就不生成二维码

项目哪里用到了分布式事务

用户支付订单，保证订单和库存，商品的一致性

解决状态最终一致性

最大努力通知
发送mq消息，需要ack确认

怎么保障消息可靠性，

生产者100%投递:设置confirm模式或return模式，重写confirmCallBack,ReturnCallback方法,confirm中的boolean值表示是否传递到交换机,然后if里面处理
mq持久化 设置 durable
消费者100%消费:手动确认消息,消息确认失败，重新发送，根据消息的级别，设置失败次数，记录日志

或者阿里巴巴seata

消息队列解决哪些问题

  1. 异步: 写数据库->发短信->更新缓存->返回结果 —- 写数据库并发送消息队列返回结果–消息队列发送通知进行缓存
  2. 并行: 写数据库并发送消息，可以同时发送短信通知消息和更新缓存信息
  3. 解耦: 将短信和使用短信的业务解耦
  4. 排队: 削峰，也能保证消息的顺序处理

mq基本概念

Broker:一个mq服务
Exchange:交换机
Queue:队列
Binding:绑定
Routing_Key=交换机的投递规则
vhost:虚拟主机，一个broker可以有多个vhost
channel:信道，一个channel代表一个会话任务

延迟队列关闭过期订单

延迟消息有两种实现方案：
1，基于死信队列
2，集成延迟插件

消息重复问题

保证消息的幂等性，使用redis自增，或者使用set，把消息存进去，如果不存在就是没重复

在数据库中设置唯一性索引，将消息id作为唯一索引，数据库插入失败，回滚

秒杀模块

秒杀的数据怎么来的

后台有秒杀商品录入模块

定时任务->凌晨发消息给秒杀服务->以场次(时间段为key)，商品id为hash里面的key，商品详情为value，添加到redis里

发布消息，通知各个服务器，商品已发布，消息内容 skuId:1, 1代表有商品

网关限流

令牌桶算法

在 Spring Cloud Gateway 上实现限流是个不错的选择，只需要编写一个过滤器就可以了。有了前边过滤器的基础，写起来很轻松。
Spring Cloud Gateway 已经内置了一个RequestRateLimiterGatewayFilterFactory，我们可以直接使用。
目前RequestRateLimiterGatewayFilterFactory的实现依赖于 Redis，所以我们还要引入spring-boot-starter-data-redis-reactive。
filters:

  • name: RequestRateLimiter #filter名称必须是RequestRateLimiter
    args:
    redis-rate-limiter.replenishRate: 2 #允许用户每秒处理多少个请求
    redis-rate-limiter.burstCapacity: 2 #令牌桶的容量，允许在一秒钟内完成的最大请求数
    key-resolver: “#{@ipKeyResolver}”

项目的开发流程

1.创建一个项目
2.开启第一个迭代
3.负责人负责维护，更新，分配给开发人员需求，制定优先级，处理人，截止日期，需要描述，验收规则
4.每一个需求都用进度 待处理，处理中，已关闭
4.测试人员在缺陷模块统一反馈缺陷问题，类型等等,开发人员处理解决问题
