商品管理

对商品专门独立为一个数据中心，集中做各个业务平台的商品信息管理。

SKU是最小库存单元，SPU是标准化产品单元。
以iPhone7 Plus(SPU)为例，这个SPU的规格有多种（颜色包含金色、白色、黑色、玫瑰金、银色、亮黑、红色这6种；内存包含32G、128G、256G这3种），
对应18（即6乘以3）种SKU。比如“iPhone 7 Plus 白色 32G”、“iPhone 7 Plus 黑色 32G”

我主要负责的是sku的各项功能，比如商品分类，库存管理,商品的报损报溢，商品的上下架等，还有其他的后台管理比如订单管理,售后管理，促销管理。

项目是一个接口返回的还是前端异步调你后台多个接口返回的skuInfo、skuImage等等

使用的是前后端分离，通过一个接口返回map对象，map中包含了商品信息，商品的图片集合，商品分类等，所以通过异步编排优化实现。

那这个异步编排怎么理解呢，如何使用的在项目中？

由于商品详情展示的数据需要要调用多个查询接口，为了提高响应速度，我们采用线程异步编排的方式进行接口调用。

异步编排的难点在哪？

异步了一个线程，我执行完之后，怎么通知它的子线程，该主线程执行完成了，通知子线程拿到结果去处理

线程池中某一个线程失败了，不可能让用户等20s一个详情页面，如何处理？

异步编排通过exceptionally,能过获取异常，并且修改返回数据

向一个线程池中仍五个线程，每一个线程都会调用addWorker方法，其他一个线程发生了异常,线程池会把这个异常remove掉，然后在addWorker

那你们怎么区分es的分片和副本？

一个索引可以存储超出单个节点硬件限制的大量数据。比如，一个具有 10 亿文档数据
的索引占据 1TB 的磁盘空间，而任一节点都可能没有这样大的磁盘空间。或者单个节点处
理搜索请求，响应太慢。
为了解决这个问题，Elasticsearch 提供了将索引划分成多份的能力，
每一份就称之为分片。当你创建一个索引的时候，你可以指定你想要的分片的数量。每个分
片本身也是一个功能完善并且独立的“索引”
1）允许你水平分割 / 扩展你的内容容量。
2）允许你在分片之上进行分布式的、并行的操作，进而提高性能/吞吐量。
至于一个分片怎样分布，它的文档怎样聚合和搜索请求，是完全由 Elasticsearch 管理的，

副本（Replicas）
在一个网络 / 云的环境里，失败随时都可能发生，在某个分片/节点不知怎么的就处于
离线状态，或者由于任何原因消失了，这种情况下，有一个故障转移机制是非常有用并且是
强烈推荐的。为此目的，Elasticsearch 允许你创建分片的一份或多份拷贝，这些拷贝叫做复
制分片(副本)。
1.在分片/节点失败的情况下，提供了高可用性。因为这个原因，注意到复制分片从不与
原/主要（original/primary）分片置于同一节点上是非常重要的。
2.扩展你的搜索量/吞吐量，因为搜索可以在所有的副本上并行运行。

那你们分片和副本一般怎么设置呢，设置的原则是什么呢？

三台es服务，六个分片，三个主分片，三个副本
因为写索引时，同时也会向副本写入，副本节点越多，写索引效率越慢

es调优参数

如果cpu使用率很高，将索引最大线程数设置为1，有效调节写入性能，因为并发写需要寻址，写入性能不会提升
refresh,调整数据写入后多少秒能被搜索到，默认是1s，如果对实时性要求不大，将这个参数调高，能提升性能

商品详情模块

商品详情的原始设计

当我们点开商品详情页面的时候，要通过商品id查询商品信息，商品实时价格，商品图片，商品库存，商品分类，商品的销售属性等，每一个功能都对应这不同的表，这样就产生两个问题
1.数据库访问压力大
2.串行化代码执行时间长，效率低

解决方案

  1. 使用异步编程CompleteFuture，CompleteFuture是基于Future的增强，他运行在与主线程不同的线程上，能够获取主线程的进度，完成或者失败，再编写非阻塞代码，这样就能实现并行的执行多个请求，提高程序的性能
    CompletableFuture的supplyAsync(查询skuinfo信息，自定义线程池)
    supplyAsync()有返回值（返回值是泛型）。由于返回值是泛型，所以可以任意return，返回用户的信息，如果用户id不为空
    并行执行5个接口，采用thenRunAsync，将每个任务查询到的结果存储到一个ConcurrentHashMap中
    使用CompletableFuture.allOf().join()，在所有任务执行完以后,将ConcurrentHashMap集成的所有值返回
  2. redis优化
    mysql默认几千并发，redis单机就能有上万并发，所有将商品数据进行预热,因为要面对三个问题
    缓存穿透：指redis和数据库都没有数据，每次请求都会查询数据库，如果同一时间大量请求访问数据库，就会给数据库造成巨大压力
    解决方案：设置空的对象，30秒的过期时间
    缓存击穿：一个热门的key突然失效，大量的并发访问数据库
    解决方案：使用redis分布式锁，调用redissonClient的getLock方法，传入锁的名称,本质是redis的setnx,redission的unlock方法本质就是uuid和lua表达式
    setnx：如果key（锁）存在就set失败，同时，哪个线程set成功哪个线程就返回true，相当于拿到了锁，接着就实现业务逻辑，在业务逻辑实现完成后就del键值对，其他线程再抢锁（因为其他线程一直都在尝试setnx)
    缓存雪崩：缓存雪崩是指缓存中大批量的数据同时过期，巨大的请求量直接落到 db 层，引起 db 压力过大甚至宕机
    解决方案：预防大面积的 key 同时失效，可以给不同的 key 过期时间加上随机值，让缓存失效的时间点尽量均匀，这样可以保证数据不会在同一时间大面积失效。

用户信息

在登录之后，前端可以获取到我所返回的jwt，然后通过它保持登录状态和获取信息。然后在调用接口的时候将信息传递给我进行各种操作。
我们需要很简便地在执行操作的过程中，在任何一个方法中都能获取到当前用户的一些基本信息。
创建ThreadLocal类，创建拦截器，解析荷载中的token，通过ThreadLocal.set保存,通过ThreadLocal.get获取
ThreadLocal 的作用是提供线程内的局部变量,ThreadLocal为变量在每个线程中都创建了一个副本，那么每个线程可以访问自己内部的副本变量。

ThreadLocal原理

set方法，首先获取到了当前线程t，然后调用getMap获取ThreadLocalMap，
如果map存在，则将当前线程对象t作为key，要存储的对象作为value存到map里面去。如果该Map不存在，则初始化一个。

get方法：调用当期线程t，返回当前线程t中的成员变量threadLocals。而threadLocals其实就是ThreadLocalMap。

在进行get之前，必须先set，否则会报空指针异常，当然也可以初始化一个，但是必须重写initialValue()方法。

ThreadLocal内存泄漏

  1. Thread中有一个map，就是ThreadLocalMap
    2、ThreadLocalMap的key是ThreadLocal，值是我们自己设定的。
    3、ThreadLocal是一个弱引用，当为null时，会被当成垃圾回收
    4、重点来了，突然我们ThreadLocal是null了，也就是要被垃圾回收器回收了，但是此时我们的ThreadLocalMap生命周期和Thread的一样，它不会回收，这时候就出现了一个现象。那就是ThreadLocalMap的key没了，但是value还在，这就造成了内存泄漏。
    解决办法：使用完ThreadLocal后，执行remove操作，避免出现内存溢出情况

线程的副本是什么

多线程并发之所以会不安全，就是因为线程不拥有资源，它们共享竞争着进程的资源，这样线程并发起来不安全，一般的解决方案便是用锁，保证每时每刻一个资源最多只能被一个线程拥有。
而在java的多线程中，ThreadLocal为解决多线程程序的并发问题提供了一种新的思路。
ThreadLocal解决多线程的并发问题的思路很简单：就是为每一个线程维护一个副本变量，从而让线程拥有私有的资源，就不再需要去竞争进程中的资源。每一个线程都可以独立地改变自己的副本，而不会影响其它线程所对应的副本。

使用流式处理时，在数据量较大的情况下，通过并行流可以开启多个线程来执行处理，

所以当对方法替换的同时，必须关注方法处理的过程中，

当变量在方法外部定义尤为重要，使用并行流需谨慎，时刻考虑线程安全问题。否则可能造成程序死锁，或数据的准确性。造成的后果完全取决于使用非线程安全类的效果。

AtomicInteger原理：

get(),set():Atomic的变量都是volatile修饰的，直接从内存取值,彼此可见
AddAndGet 和 getAndIncrement、getAndAdd、incrementAndGet 等等方法都是使用了 do … while + CAS 操作，
其实也就相当于是一个自旋锁，如果 CAS 修改成功就会一直循环，修改失败才会返回

CAS

AtomicInteger 中有两个方法 get 和 set 吗？它们只是用 volatile 修饰了一下，而 volatile 不具有原子性,
一种是使用synchronized 和 lock 等类似的加锁机制，这种锁具有独占性，也就是说同一时刻只能有一个线程来进行修改，这种方式能够保证原子性，但是相对开销比较大，这种锁也叫做悲观锁。
另外一种解决办法是使用版本号或者是 CAS 方法。

CAS 有三个操作数：当前值A、内存值V、要修改的新值B
假设 当前值A 跟 内存值V 相等，那就将 内存值V 改成B
假设 当前值A 跟 内存值V 不相等，要么就重试，要么就放弃更新

解决aba问题

给变量加一个版本号即可，在比较的时候不仅要比较当前变量的值 还需要比较当前变量的版本号。
在java5中，已经提供了AtomicStampedReference来解决问题，检查当前引用是否等于预期引用，其次检查当前标志是否等于预期标志，如果都相等就会以原子的方式将引用和标志都设置为新值

版本号

版本号机制是在数据表中加上一个 version字段来实现的，
当执行写操作并且写入成功后，version = version + 1，
当线程 A 要更新数据时，在读取数据的同时也会读取 version 值，
在提交更新时，若刚才读取到的 version 值为当前数据库中的 version 值相等时才更新，
否则重试更新操作，直到更新成功

扣减库存以及清空购物车

  1. 扣减库存会出现超卖问题：原因是因为查询库存和扣减存储之间有时间差，利用mysql的乐观锁解决，在修改的时候添加库存大于扣减的where条件就行，但这是并发很小，只有一两千，可以采用主从复制，主库修改库存，从库读取库存
  2. 会出现重复下单问题：解决方案是通过redis的自增，新增订单的时候自增为1，如果标识位>1,此时抛出重复提交订单异常,设置redis的key是用户名加商品信息表单的加密值， 在完成支付，取消订单，下单失败时删除自增值
  3. 取消订单功能：取消订单分为主动取消和超时取消，超时取消的实现方式是使用延迟队列（ttl(过期时间） + 死信队列),发送订单消息，设置过期时间为30分钟，如果30分钟后，状态为未支付，就取消订单

下单流程升级

同步排队，异步下单，当用户下单时，不需要传统流程中的校验，查询，下单，而是直接同步返回下单成功，然后异步将下单消息发送到RabbitMQ，通过消息的消费者，异步完成校验与下单，
通过消息队列的先到先得的规则，实现公平队列，以及流量削峰。下单成功的用户就返回付款界面，下单失败的用户返回消息中处理的异常,比如库存不足之类的
通过redis的List队列实现,构建和库存数量相同的队列，下单就rightPop,回滚库存就leftPush，当队列为空就表示售罄。

采用的是下单减库存的方式，为了防止用户恶意提交订单，通过Gateway网关实现令牌桶限流,限制一个ip的用户提交次数，同时设置黑名单，

解决幂等性

订单有效期是多久，怎么取消订单？

订单有效期为30分钟，我们采用 rabbitmq 的延时队列，在生成订单时，发送延时消息，设置消息30分钟后触发，然后监听到这个消息后，进行订单的取消。

怎么保证消息队列的顺序消费

一个queue，有多个consumer去消费，这样就会造成顺序的错误，consumer从MQ里面读取数据是有序的，但是每个consumer的执行时间是不固定的，无法保证先读到消息的consumer一定先完成操作，
这样就会出现消息并没有按照顺序执行，造成数据顺序错误。
解决方案
拆分多个queue，每个queue一个consumer，就是多一些queue而已，确实是麻烦点；这样也会造成吞吐量下降，可以在消费者内部采用多线程的方式取消费。

如何避免消息队列消息重复消费(如何保证消息的幂等性)

1.生产者丢失消息
使用confirm模式,在生产者那里设置开启 confirm 模式之后，生产者每次写的消息都会分配一个唯一的 id，如果消息成功写入 RabbitMQ 中，
RabbitMQ 会给生产者回传一个 ack 消息， 告诉你说这个消息 ok 了。
如果 RabbitMQ 没能处理这个消息，会回调你的一个 nack 接口，告诉你这个消息接收失败，生产者可以发送。
2.mq丢失消息,
mq接受到生产者的消息，还没被消费者消费完，就宕机了
解决方案是开启rabbitmq的持久化，同时配置你队列和消息设置的持久化，rabbitmq就会将消息持久化
3.消费者丢失消息
rabbitmq默认会自动通过ack确认成功消费的消息

怎么防止重复消费

先说为什么会重复消费：正常情况下，消费者在消费消息的时候，消费完毕后，会发送一个确认消息给消息队列，消息队列就知道该消息被消费了，就会将该消息从消息队列中删除；但是因为网络传输等等故障，确认信息没有传送到消息队列，导致消息队列不知道自己已经消费过该消息了，再次将消息分发给其他的消费者。
消息生产的时候，每一条消息都有自带的消息id，作为幂等性和去重的依据，避免重复的消息进入队列
消息消费的时候，使用redis自增啊，一般消费信息中，通过信息的id，比如说唯一的商品id，通过redis的incr判断是否已经被消费过了

什么是面向对象？

传统的面向对象洗衣服 放洗衣粉，放衣服 执行洗衣机洗衣服，执行烘干,将他们抽象成两个对象，人和洗衣机，然后执行方法,这就是面向对象。
面向过程与面向对象的优缺点：面向过程和面向对象就像是蛋炒饭和盖浇饭，盖浇饭实现了饭菜分离，提高了灵活性，饭不满意就换饭，菜不满意就换菜，可维护性，饭菜之间的耦合度更低。
缺点的话，类的调用需要实例化，更消耗资源，在嵌入式开发，单片机中一般采用面向过程，性能更重要。
面向对象有三大特性：封装，继承，多态
封装，封装把一个对象的属性私有化，同时提供一些可以被外界访问的属性的方法。
继承：继承是使用已存在的类的定义作为基础建立新类的技术，新类的定义可以增加新的数据或新的功能，也可以用父类的功能，但不能选择性地继承父类。
通过使用继承我们能够非常方便地复用以前的代码，能够大大的提高开发的效率。
多态：Java 实现多态有 3 个必要条件：继承、重写和向上转型。只有满足这3个条件，开发人员才能够在同一个继承结构中使用统一的逻辑实现代码处理不同的对象，从而执行不同的行为。
多态的体现就是，我们可以用接口类型去new出所有实现该接口的子类对象。
所以我个人认为多态更多的就是一种开发上的权宜之计，因为我们在编写一个统一的接口时，考虑不到那么多，只能大致的预留一个参数，这个参数当时可能是一个单独的类，也可能已经是一些子类的父类。可是在当时的情况下，我还并不知道这个项目后期会出现哪些重大的变化，会增加哪些新的功能？也许，随着项目的壮大，会出现越来越多的子类，我设计的方法也可能在某一天无法满足项目的需求。

SpringBean的生命周期

  1. 通过扫描注解，xml，配置等，需要被Spring管理的信息。
  2. 随后将这些信息封装成BeanDefinition，放到BeanDefinitionMap中，key是beanName，value是BeanDefinition，这里是吧元数据加载，还没有实例化对象。
  3. 接着我们调用Bean工厂后置处理器(BeanFactoryPostProcess),获取Bean对象。
  4. 在Spring中是通过反射，选择构造器实例化(Instantiation)这个Bean对象。这时Bean的实例化就完成了。
  5. 接着是初始化对象，首先看是否实现了Aware(感知)接口，接口内定义了一个set方法,比如BeanNameAware，通过set方法，可以获取和修改Bean的名字。
  6. 然后调用Bean后置处理器，里面有After和，InitializationBean,Before方法,分别是Bean初始化之前之后添加方法，Spring的AOP就是基于这个实现的
  7. 中间进行了InitializationBean的init-method方法，初始化Bean对象。
  8. 此时我们就能够获取这个Bean对象了，Bean默认是单例的，交由IOC容器管理，单例的话，节省创建和GC的内存开销，Spring大多数时候管理的都是无状态的对象，比如说VO(视图对象），DTO(数据传输对象）,DO（领域对象，抽取出的业务实体）,PO(持久化对象),
    还有Service、DAO和Controller，这些对象并没有自己的状态，它们只是用来执行某些操作的。例如，每个DAO提供的函数都只是对数据库的CRUD,
  9. 无状态就是不会保存数据,而且多线程调用一个实例的方法，因为每个线程都有自己的调用栈，局部变量保存在线程各自的调用栈里，不会共享，自然也就不存在并发问题。

    对线程的理解

    在多个线程访问某个方法或对象的时候，线程交替执行，在程序中没有同步干预的情况下，这个方法的执行或修改都能按照预期的结果来反馈，那么这个类我们认为是线程安全的。
    实际上，线程安全的类有三个特性，也就是原子性，有序性，可见性。

原子性 Java的一条语句往往需要多条 CPU 指令完成(i++)，由于操作系统的线程切换很可能导致 i++ 操作未完成，其他线程“中途”操作了共享变量 i ，导致最终结果并非我们所期待的。
有序性 cpu优化对指令重排序，导致程序编写的顺序和cpu运行顺序不一致，所以有序性会导致可见性问题,可以用Volatile关键字来解决
可见性 在多线程环境中，读和写可能是发生在不同的线程中的，有可能出现某个线程对对象的修改,对其他线程不是实时可见的，导致可见性的原因是cpu的高速缓存，l1，l2缓存不共享
要判断一个线程是否安全，就要看有没有多个线程同时访问一个共享变量,像spring中，只要不涉及操作同一个成员变量，就很少考虑线程安全问题

解决线程安全问题的几个思路：
保证代码的原子性：使用sync或者lock包
保证操作的原子性，atomic包下的类,比如AtomicInteger,AtomicReference,原子更新引用类型;}
保证操作的可见性：多线程通过内存访问到同一变量，考虑volatile
如果是集合：使用concurrent包

死锁

什么是死锁

一组互相竞争资源的线程，因为互相等待，导致永久阻塞的现象

发生死锁的条件有四个

1.互斥条件：共享资源x和y只能被一个线程所占用
2.占有且等待：线程t1已经获得了共享资源x，并且想要获取另一个资源，而该资源被其他进程所占有。
3.不可抢占：其他线程不能强行抢占线程t1占有的资源，资源只能再t1完成后自愿释放
4.循环等待：线程t1等待线程t2占有的资源，线程t2等待线程t1占有的资源

如何打破死锁

我们只需要打破任意一个条件就能避免死锁，第一个条件，互斥条件是无法破坏的，锁本身就是通过互斥来解决线程安全问题的
其他三个条件，
对于占有且等待，我们一次性申请获取所有的资源，这样就不用等待了
第二个是不可抢占，占有t1资源的线程如果去申请t2资源的时候，如果申请不到，可以主动释放t1资源
第三个是循环等待，可以通过按序申请，资源是有线性顺序的，按照顺序申请就不存在循环等待了

Redis的线程安全问题

我会从两方面回答，
redis本身是线程安全的数据库，也就是说在redis server中执行命令的时候，不存在任何的同步机制，也不会存在任何的线程安全问题.
虽然redis6.0增加了多线程，但只是用来处理网络的io事件，指令的执行过程任何是主线程处理的。

为什么Redis没有采用多线程的执行

多线程的开销，在不够良好的系统设计下，采用多线程后，刚开始增加线程数，系统吞吐率会增加，再增加线程数，系统吞吐率就会增长延迟了,甚至可能下降
1.比如i7-12700k,硬件线程有20个，通常情况下软件线程数量大于硬件线程，于是有了线程的调度，会通过保存和恢复线程缓存，cpu寄存器，影响执行性能。
2.多线程为了保证共享资源的正确性，通常会加锁

Redis为什么使用了单线程那么快

1.Redis的操作是通过内存完成的
2.采用了高效的数据结果，比如哈希表和跳表
3.采用了多路复用机制，Redis采用了Linux的IO多路复用，改机制运行内核中同时存在多个监听套接字和已连接套接字,这样Redis就能同时和多个客户端连接并处理请求

GET 和 POST 的区别？

1.get是向服务器索取数据，post是提交请求
2.get将参数绑定是url中，通过？传参，通过&表明不同的参数,post是把参数放到http请求体中
3.get的参数是可见的，post不可见
4.get发送一次tcp数据包，post发送两次tcp数据包
对于GET方式的请求，浏览器会把http header和data一并发送出去，服务器响应200（返回数据）

他首先对请求进行预检，判断是否能进行跨域请求
实际使用时
get：只为获取数据，同时加上请求参数的url不会特别大，且没有需要保密的字段，则用get请求；
post：为了提交数据，比如提交一些数据给后台保存、删除等，无论是否有一定数据量均可使用post；另外，使用post提交请求来get数据也是可以的。

什么是分布式事务

分布式事务指事务的参与者、支持事务的服务器、资源服务器以及事务管理器分别位于不同的分布式系统的不同节点之上,且属于不同的应用。
指一次大的操作由不同的小操作组成的，这些小的操作分布在不同的服务器上，分布式事务需要保证这些小操作要么全部成功，要么全部失败。

什么原因导致Full gc呢？

1.系统承载高并发请求，或者处理数据量过大，导致Young GC很贫乏，而且每次Young GC过后存活对象太多，
内存分配不合理，Survivor区过小，导致对象频繁进入老年代，频繁触发Full GC。
2.系统一次性加载过多数据进内存，搞出来很多大对象，导致频繁有大对象进入老年带，必然频繁触发Full GC。
3.系统发生了内存泄漏，莫名其妙创建大量的对象，始终无法回收，一直占用在老年代里，必然频繁触发Full GC

我们知道AOP实际上是Spring在运行时创建了一个代理对象，也就是说这个对象，是我们在运行时创建的，而不是一开始就定义好的，这很符合工厂方法模式。更形象地说，AOP代理对象通过Java的反射机制，在运行时创建了一个代理对象，在代理对象的目标方法中根据业务要求织入了相应的方法。这个对象在Spring中就是——ProxyFactoryBean。

你了解HTTPS的过程吗？

HTTPS在我的理解下，就是「安全」的HTTP协议（客户端与服务端的传输链路中进行加密）
服务端去认证的CA机构去、申请一份数据证书，证书里面包含证书持有者，证书有效期，公钥
ca机构自己也有一份公钥，在发布证书前，通过私钥给证书加密
当客户端请求服务器，服务器返回证书给客户端，客户端通过公钥（ca是公信机构，每个浏览器里面都有ca的公钥），这时候就能判断证书是否可信
解决了通信真实性的问题
其次要解决通信加密的问题
客户端通过服务端发送的数字证书，获取里面的公钥
客户端生成一个key作为对称加密的私钥，通过服务端的公钥发送过去
服务端用自己的私钥解密服务端公钥，得到对称加密的密钥
之后就能够使用对称加密的密钥快乐的通信加密

为什么数组的下标从0开始不是从1开始？

假设一个int数据,长度为4，a[0] = 1000-1003,a[1] = 1004-1007,那么首地址就是1000，当我们想要访问一个元素，采用寻址公式
a[i]地址= 首地址+i*data_type_size,数据的下标准确的说叫偏移(offset),如果从1开始，那么每一次访问元素，都要多执行一次-1的运算，性能就消耗一些。

在商品详情页使用分布式锁不是会慢吗？

不会，解决缓存击穿，第一个用户线程访问1号商品详情页时，只要1号商品这个热点key失效了，就会缓存击穿去访问到数据库的线程并且加分布式锁，
然后马上就会把1号商品相关页面的数据放进缓存，那其他后面的线程访问1号商品还是先判断缓存有没有，有就走缓存，所以一定程度避免了一定数量的线程直接访问数据库，不会慢
分布式锁概念：同一时刻保证只有一个进程在执行

mq消费方异常，消费不了消息怎么办，造成消费端消息堆积？

核心思想：紧急临时扩容，更快的速度去消费数据

  • 修复Consumer不消费问题，使其恢复正常消费，根据业务需要看是否要暂停
  • 增加消费者的多线程处理（缩短线程休眠时间等）、部署多个消费者
  • 直到堆积的消息处理完成，再还原到正常的机器数量

索引的设计原则

1.适合索引的列出现在where子句中，或者连接指定子句中指定的列
2.基数较小的表，没必要创建索引
3.创建的索引，越短越好，可以指定列的某一部分
4.不要给每一个字段都创建索引
5.定义有外键的数据列一定要创建索引
6.频繁更新的字段不要加索引

volatile 关键字的作用？

violatile的作用是禁止指令重排，保证他的原子性。
在java内存模型中，A线程和B线程拿到的都是你的共享变量副本,而不是主内存的共享变量。
a线程修改了副本中的变量参数，然后刷新会主内存中，b线程并不知道主内存数据已经更新，所有这个操作就是不可见的。
所以需要一种机制，一旦主内存修改，就通知所有的共享变量副本，保证可见性，我们常用happen-before（先行发生原则）来描述可见性。
当使用volatile修饰变量时，会在写操作后面添加一条屏障指令，一旦你写操作完成，其他线程都从主内存中读取最新值。

as if serial 单线程无论怎么样进行指令重排，单线程的结果都是最终一致的，编译器不会对数据依赖的操作进行操作重排序，但是多线程打乱了数据依赖的操作顺序。

常用的集合类型

在collection包下有三大集合，list，set，map
我最常用的是ArrayList和Hashmap
Array是基于索引实现的动态数据，那Java本身就有数组了，为什么要用ArrayList呢？原生的数组会有一个特点：你在使用的时候必须要为它创建大小，而ArrayList不用。
在日常开发的时候，往往我们是不知道数组的大小的
而ArrayList不一样，ArrayList我们在使用的时候可以往里边添加20个，30个，甚至更多的元素
因为ArrayList是实现了动态扩容的
当我们new ArrayList()的时候，默认会有一个空的Object数组，大小为0。
当我们第一次add添加数据的时候，会给这个数组初始化一个大小，这个大小默认值为10
使用ArrayList在每一次add的时候，它都会先去计算这个数组够不够空间
如果空间是够的，那直接追加上去就好了。如果不够，那就得扩容
那怎么扩容？一次扩多少？
在源码里边，有个grow方法，每一次扩原来的1.5倍。比如说，初始化的值是10嘛。
现在我第11个元素要进来了，发现这个数组的空间不够了，所以会扩到15
空间扩完容之后，会调用arraycopy来对数组进行拷贝

是由底层的数据结构来决定的，在日常开发中，遍历的需求比增删要多，即便是增删也是往往在List的尾部添加就OK了。
像在尾部添加元素，ArrayList的时间复杂度也就O(1),另外的是，ArrayList的增删底层调用的copyOf()被优化过,现代CPU对内存可以块操作，ArrayList的增删一点儿也不会比LinkedList慢

第二个常用的就是HashMap

put hashmap
他是基于键值对存储的一个集合,HashMap底层是 数组+链表+红黑树，为什么要用这几类结构呢？
其中数组的类型是Entry类型的数组，当我们往HashMap内填充元素时，首先会计算其key的hashcode来重新计算key的hash值，并找到数组中对应的下标，如果该位置目前并没有元素，则直接将该元素放入数组中；如果该位置目前已经有了元素，则把新的元素加入到链表中
使用这种存储方式是为了解决哈希碰撞的问题，换言之，链表中的每个key，都具有相同的哈希值。以便提高查找效率

new hashmap
当你new一个HashMap的时候，会发生什么吗？
HashMap有几个构造方法，但最主要的就是指定初始值大小和负载因子的大小。
如果我们不指定，默认HashMap的大小为16，负载因子的大小为0.75
HashMap的大小只能是2次幂的，假设你传一个10进去，实际上最终HashMap的大小是16，你传一个7进去，HashMap最终的大小是8，具体的实现在tableSizeFor可以看到。
我们把元素放进HashMap的时候，需要算出这个元素所在的位置（hash）。
在HashMap里用的是位运算来代替取模，能够更加高效地算出该元素所在的位置。
为什么HashMap的大小只能是2次幂，因为只有大小为2次幂时，才能合理用位运算替代取模。
