那你们怎么区分es的分片和副本？

一个索引可以存储超出单个节点硬件限制的大量数据。比如，一个具有 10 亿文档数据
的索引占据 1TB 的磁盘空间，而任一节点都可能没有这样大的磁盘空间。或者单个节点处
理搜索请求，响应太慢。
为了解决这个问题，Elasticsearch 提供了将索引划分成多份的能力，
每一份就称之为分片。当你创建一个索引的时候，你可以指定你想要的分片的数量。每个分
片本身也是一个功能完善并且独立的“索引”
1）允许你水平分割 / 扩展你的内容容量。
2）允许你在分片之上进行分布式的、并行的操作，进而提高性能/吞吐量。
至于一个分片怎样分布，它的文档怎样聚合和搜索请求，是完全由 Elasticsearch 管理的，

副本（Replicas）
在一个网络 / 云的环境里，失败随时都可能发生，在某个分片/节点不知怎么的就处于
离线状态，或者由于任何原因消失了，这种情况下，有一个故障转移机制是非常有用并且是
强烈推荐的。为此目的，Elasticsearch 允许你创建分片的一份或多份拷贝，这些拷贝叫做复
制分片(副本)。
1.在分片/节点失败的情况下，提供了高可用性。因为这个原因，注意到复制分片从不与
原/主要（original/primary）分片置于同一节点上是非常重要的。
2.扩展你的搜索量/吞吐量，因为搜索可以在所有的副本上并行运行。

那你们分片和副本一般怎么设置呢，设置的原则是什么呢？

一个分片一般最大最好限制在50g之内，不然就会影响索引的查询速度，但是呢，50都能存上2,30亿个索引了，我们的数据就只有十几万，所用没有采用分片。
因为写索引时，同时也会向副本写入，副本节点越多，写索引效率越慢

es调优参数

如果cpu使用率很高，将索引最大线程数设置为1，有效调节写入性能，因为并发写需要寻址，写入性能不会提升
refresh,调整数据写入后多少秒能被搜索到，默认是1s，如果对实时性要求不大，将这个参数调高，能提升性能

2.串行化代码执行时间长，效率低

解决方案

  1. 使用异步编程CompleteFuture，CompleteFuture是基于Future的增强，他运行在与主线程不同的线程上，能够获取主线程的进度，完成或者失败，再编写非阻塞代码，这样就能实现并行的执行多个请求，提高程序的性能
    CompletableFuture的supplyAsync(查询skuinfo信息，自定义线程池)
    supplyAsync()有返回值（返回值是泛型）。由于返回值是泛型，所以可以任意return，返回用户的信息，如果用户id不为空
    并行执行5个接口，采用thenRunAsync，将每个任务查询到的结果存储到一个ConcurrentHashMap中
    使用CompletableFuture.allOf().join()，在所有任务执行完以后,将ConcurrentHashMap集成的所有值返回
  2. redis优化
    mysql默认几千并发，redis单机就能有上万并发，所有将商品数据进行预热,因为要面对三个问题
    缓存穿透：指redis和数据库都没有数据，每次请求都会查询数据库，如果同一时间大量请求访问数据库，就会给数据库造成巨大压力
    解决方案：设置空的对象，30秒的过期时间
    缓存击穿：一个热门的key突然失效，大量的并发访问数据库
    解决方案：使用redis分布式锁，调用redissonClient的getLock方法，传入锁的名称,本质是redis的setnx,redission的unlock方法本质就是uuid和lua表达式
    setnx：如果key（锁）存在就set失败，同时，哪个线程set成功哪个线程就返回true，相当于拿到了锁，接着就实现业务逻辑，在业务逻辑实现完成后就del键值对，其他线程再抢锁（因为其他线程一直都在尝试setnx)
    缓存雪崩：缓存雪崩是指缓存中大批量的数据同时过期，巨大的请求量直接落到 db 层，引起 db 压力过大甚至宕机
    解决方案：预防大面积的 key 同时失效，可以给不同的 key 过期时间加上随机值，让缓存失效的时间点尽量均匀，这样可以保证数据不会在同一时间大面积失效。

ThreadLocal 的作用是提供线程内的局部变量,ThreadLocal为变量在每个线程中都创建了一个副本，那么每个线程可以访问自己内部的副本变量。

ThreadLocal原理

set方法，首先获取到了当前线程t，然后调用getMap获取ThreadLocalMap，
如果map存在，则将当前线程对象t作为key，要存储的对象作为value存到map里面去。如果该Map不存在，则初始化一个。

get方法：调用当期线程t，返回当前线程t中的成员变量threadLocals。而threadLocals其实就是ThreadLocalMap。

在进行get之前，必须先set，否则会报空指针异常，当然也可以初始化一个，但是必须重写initialValue()方法。

ThreadLocal内存泄漏

  1. Thread中有一个map，就是ThreadLocalMap
    2、ThreadLocalMap的key是ThreadLocal，值是我们自己设定的。
    3、ThreadLocal是一个弱引用，当为null时，会被当成垃圾回收
    4、重点来了，突然我们ThreadLocal是null了，也就是要被垃圾回收器回收了，但是此时我们的ThreadLocalMap生命周期和Thread的一样，它不会回收，这时候就出现了一个现象。那就是ThreadLocalMap的key没了，但是value还在，这就造成了内存泄漏。
    解决办法：使用完ThreadLocal后，执行remove操作，避免出现内存溢出情况

线程的副本是什么

多线程并发之所以会不安全，就是因为线程不拥有资源，它们共享竞争着进程的资源，这样线程并发起来不安全，一般的解决方案便是用锁，保证每时每刻一个资源最多只能被一个线程拥有。
而在java的多线程中，ThreadLocal为解决多线程程序的并发问题提供了一种新的思路。
ThreadLocal解决多线程的并发问题的思路很简单：就是为每一个线程维护一个副本变量，从而让线程拥有私有的资源，就不再需要去竞争进程中的资源。每一个线程都可以独立地改变自己的副本，而不会影响其它线程所对应的副本。

AtomicInteger原理：

get(),set():Atomic的变量都是volatile修饰的，直接从内存取值,彼此可见
AddAndGet 和 getAndIncrement、getAndAdd、incrementAndGet 等等方法都是使用了 do … while + CAS 操作，
其实也就相当于是一个自旋锁，如果 CAS 修改成功就会一直循环，修改失败才会返回

CAS

AtomicInteger 中有两个方法 get 和 set 吗？它们只是用 volatile 修饰了一下，而 volatile 不具有原子性,
一种是使用synchronized 和 lock 等类似的加锁机制，这种锁具有独占性，也就是说同一时刻只能有一个线程来进行修改，这种方式能够保证原子性，但是相对开销比较大，这种锁也叫做悲观锁。
另外一种解决办法是使用版本号或者是 CAS 方法。

CAS 有三个操作数：当前值A、内存值V、要修改的新值B
假设 当前值A 跟 内存值V 相等，那就将 内存值V 改成B
假设 当前值A 跟 内存值V 不相等，要么就重试，要么就放弃更新

解决aba问题

给变量加一个版本号即可，在比较的时候不仅要比较当前变量的值 还需要比较当前变量的版本号。
在java5中，已经提供了AtomicStampedReference来解决问题，检查当前引用是否等于预期引用，其次检查当前标志是否等于预期标志，如果都相等就会以原子的方式将引用和标志都设置为新值

版本号

版本号机制是在数据表中加上一个 version字段来实现的，
当执行写操作并且写入成功后，version = version + 1，
当线程 A 要更新数据时，在读取数据的同时也会读取 version 值，
在提交更新时，若刚才读取到的 version 值为当前数据库中的 version 值相等时才更新，
否则重试更新操作，直到更新成功

怎么保证消息队列的顺序消费

一个queue，有多个consumer去消费，这样就会造成顺序的错误，consumer从MQ里面读取数据是有序的，但是每个consumer的执行时间是不固定的，无法保证先读到消息的consumer一定先完成操作，
这样就会出现消息并没有按照顺序执行，造成数据顺序错误。
解决方案
拆分多个queue，每个queue一个consumer，就是多一些queue而已，确实是麻烦点；这样也会造成吞吐量下降，可以在消费者内部采用多线程的方式取消费。

如何避免消息队列消息重复消费(如何保证消息的幂等性)

1.生产者丢失消息
使用confirm模式,在生产者那里设置开启 confirm 模式之后，生产者每次写的消息都会分配一个唯一的 id，如果消息成功写入 RabbitMQ 中，
RabbitMQ 会给生产者回传一个 ack 消息， 告诉你说这个消息 ok 了。
如果 RabbitMQ 没能处理这个消息，会回调你的一个 nack 接口，告诉你这个消息接收失败，生产者可以发送。
2.mq丢失消息,
mq接受到生产者的消息，还没被消费者消费完，就宕机了
解决方案是开启rabbitmq的持久化，同时配置你队列和消息设置的持久化，rabbitmq就会将消息持久化

SpringBean的生命周期

  1. 通过扫描注解，xml，配置等，需要被Spring管理的信息。
  2. 随后将这些信息封装成BeanDefinition，放到BeanDefinitionMap中，key是beanName，value是BeanDefinition，这里是吧元数据加载，还没有实例化对象。
  3. 接着我们调用Bean工厂后置处理器(BeanFactoryPostProcess),获取Bean对象。
  4. 在Spring中是通过反射，选择构造器实例化(Instantiation)这个Bean对象。这时Bean的实例化就完成了。
  5. 接着是初始化对象，首先看是否实现了Aware(感知)接口，接口内定义了一个set方法,比如BeanNameAware，通过set方法，可以获取和修改Bean的名字。
  6. 然后调用Bean后置处理器，里面有After和，InitializationBean,Before方法,分别是Bean初始化之前之后添加方法，Spring的AOP就是基于这个实现的
  7. 中间进行了InitializationBean的init-method方法，初始化Bean对象。
  8. 此时我们就能够获取这个Bean对象了，Bean默认是单例的，交由IOC容器管理，单例的话，节省创建和GC的内存开销，Spring大多数时候管理的都是无状态的对象，比如说VO(视图对象），DTO(数据传输对象）,DO（领域对象，抽取出的业务实体）,PO(持久化对象),
    还有Service、DAO和Controller，这些对象并没有自己的状态，它们只是用来执行某些操作的。例如，每个DAO提供的函数都只是对数据库的CRUD,
  9. 无状态就是不会保存数据,而且多线程调用一个实例的方法，因为每个线程都有自己的调用栈，局部变量保存在线程各自的调用栈里，不会共享，自然也就不存在并发问题。

    对线程的理解

    在多个线程访问某个方法或对象的时候，线程交替执行，在程序中没有同步干预的情况下，这个方法的执行或修改都能按照预期的结果来反馈，那么这个类我们认为是线程安全的。
    实际上，线程安全的类有三个特性，也就是原子性，有序性，可见性。

原子性 Java的一条语句往往需要多条 CPU 指令完成(i++)，由于操作系统的线程切换很可能导致 i++ 操作未完成，其他线程“中途”操作了共享变量 i ，导致最终结果并非我们所期待的。
有序性 cpu优化对指令重排序，导致程序编写的顺序和cpu运行顺序不一致，所以有序性会导致可见性问题,可以用Volatile关键字来解决
可见性 在多线程环境中，读和写可能是发生在不同的线程中的，有可能出现某个线程对对象的修改,对其他线程不是实时可见的，导致可见性的原因是cpu的高速缓存，l1，l2缓存不共享
要判断一个线程是否安全，就要看有没有多个线程同时访问一个共享变量,像spring中，只要不涉及操作同一个成员变量，就很少考虑线程安全问题

解决线程安全问题的几个思路：
保证代码的原子性：使用sync或者lock包
保证操作的原子性，atomic包下的类,比如AtomicInteger,AtomicReference,原子更新引用类型;}
保证操作的可见性：多线程通过内存访问到同一变量，考虑volatile
如果是集合：使用concurrent包

死锁

什么是死锁

一组互相竞争资源的线程，因为互相等待，导致永久阻塞的现象

发生死锁的条件有四个

1.互斥条件：共享资源x和y只能被一个线程所占用
2.占有且等待：线程t1已经获得了共享资源x，并且想要获取另一个资源，而该资源被其他进程所占有。
3.不可抢占：其他线程不能强行抢占线程t1占有的资源，资源只能再t1完成后自愿释放
4.循环等待：线程t1等待线程t2占有的资源，线程t2等待线程t1占有的资源

如何打破死锁

我们只需要打破任意一个条件就能避免死锁，第一个条件，互斥条件是无法破坏的，锁本身就是通过互斥来解决线程安全问题的
其他三个条件，
对于占有且等待，我们一次性申请获取所有的资源，这样就不用等待了
第二个是不可抢占，占有t1资源的线程如果去申请t2资源的时候，如果申请不到，可以主动释放t1资源
第三个是循环等待，可以通过按序申请，资源是有线性顺序的，按照顺序申请就不存在循环等待了

什么是cap

什么是CAP？
Consistency一致性，Availability可用性，Partition tolerance分区容错性。
分布式系统的最大难点，就是各个节点的状态如何同步。CAP 定理是这方面的基本定理，也是理解分布式系统的起点

#Base
基本可用(Basically Available)：
基本可用是指分布式系统在出现故障的时候，允许损失部分可用性，即保证核心可用。
电商大促时，为了应对访问量激增，部分用户可能会被引导到降级页面，服务层也可能只提供降级服务。这就是损失部分可用性的体现。
软状态(Soft State)： 软状态是指允许系统存在中间状态，而该中间状态不会影响系统整体可用性。
分布式存储中一般一份数据至少会有三个副本，允许不同节点间副本同步的延时就是软状态的体现。MySQL Replication 的异步复制也是一种体现。
最终一致性(Eventual Consistency)： 最终一致性是指系统中的所有数据副本经过一定时间后，最终能够达到一致的状态。
弱一致性和强一致性相反，最终一致性是弱一致性的一种特殊情况。

什么是分区容错性

分区容错性就是将相同的数据复制到各各区域(节点中)，这样即时有些节点坏掉了，也能保障数据网络连通。
然而将数据复制到多个节点上，就会带来一致性问题，多个节点上的复制数据可能是不一致的，
要保证一致性，要每次写操作，等待所有的节点都写入成功，这个等待的过程，就是可用性问题。

Zookeeper

Zookeeper采用的是cp，采用的是zab原子广播协议，主节点接收客户瑞发送的写请求，操作完成后，向所有的从节点广播请求(请求数据同步),等待所有的从节点反馈，
在zab中，只要有半数的节点反馈ok，主节点就会向所有的从节点发送commit消息(即开始数据同步)。
Zookeeper如果主节点崩溃，从节点选举的过程中，是不能写入数据的，一般选举的时间有30s-120s，漫长的选举期，导致整个zk都是不可用的。

二阶段提交

协调者向所有参与者发送prepare请求与事务内容，询问是否可以准备事务提交，并等待参与者的响应。
参与者执行事务中包含的操作，并记录undo日志（用于回滚）和redo日志（用于重放），但不真正提交。
参与者向协调者返回事务操作的执行结果，执行成功返回yes，否则返回no。
若所有参与者都返回yes，说明事务可以提交：
协调者向所有参与者发送commit请求。
参与者收到commit请求后，将事务真正地提交上去，并释放占用的事务资源，并向协调者返回ack。
协调者收到所有参与者的ack消息，事务成功完成。
若有参与者返回no或超时未返回，说明事务需要回滚。

Zookeeper机制

1.消息广播 zookeeper采用的数据副本同步方式和二阶段提交类似，但是只要求有一半的ack确认就行，减少了二阶段提交的阻塞性问题
2.崩溃恢复 zookeeper为了保证所有进程的有序执行，只有主节点可以接受并写入请求，若主节点崩溃，则进行崩溃恢复和从节点选举。
新选举的节点不能包含未提交的事务，同时这个节点中有最高的zxid，
为了保证数据一致性，通过递增的发生生成事务id，给每一个提议都加上事务id
3.数据同步 新的节点选举成功后，将最高的zxid的事务发送给其他从节点，从节点会根据事务进行数据同步或回退，保证数据的一致性

什么原因导致Full gc呢？

1.系统承载高并发请求，或者处理数据量过大，导致Young GC很贫乏，而且每次Young GC过后存活对象太多，
内存分配不合理，Survivor区过小，导致对象频繁进入老年代，频繁触发Full GC。
2.系统一次性加载过多数据进内存，搞出来很多大对象，导致频繁有大对象进入老年带，必然频繁触发Full GC。
3.系统发生了内存泄漏，莫名其妙创建大量的对象，始终无法回收，一直占用在老年代里，必然频繁触发Full GC

volatile 关键字的作用？

violatile的作用是禁止指令重排，保证他的原子性。
在java内存模型中，A线程和B线程拿到的都是你的共享变量副本,而不是主内存的共享变量。
a线程修改了副本中的变量参数，然后刷新会主内存中，b线程并不知道主内存数据已经更新，所有这个操作就是不可见的。
所以需要一种机制，一旦主内存修改，就通知所有的共享变量副本，保证可见性，我们常用happen-before（先行发生原则）来描述可见性。
当使用volatile修饰变量时，会在写操作后面添加一条屏障指令，一旦你写操作完成，其他线程都从主内存中读取最新值。

as if serial 单线程无论怎么样进行指令重排，单线程的结果都是最终一致的，编译器不会对数据依赖的操作进行操作重排序，但是多线程打乱了数据依赖的操作顺序。

多线程关键字

1.sleep
sleep是Thread的方法，是指定线程睡眠指定时间，但是线程并不会释放锁。
举个例子，A、B两个线程，都需要获取同一个对象锁才能执行。假如A拿到了开始执行，然后调用sleep方法进入暂停状态，B线程依然无法执行，因为A即使睡着了，也死死抓紧对象锁不放。B只能等A醒过来，执行完毕自己放掉对象锁，B才能执行。
2.wait
wait是Object的方法。与sleep不同的是，一旦wait，线程就会释放锁，并且需要其他线程通过notify或notifyAll方法唤醒才行，自己不会主动醒过来。
3.yield
yield方法是让当前线程暂停执行，回到可执行状态，同样也不会释放锁。而且调用yield方法后并不一定其他线程就可以执行了，也可能该线程刚调用yield方法想让出cpu资源，下一刻该线程又被选上了，它继续执行。
4.join
join相当于是让别人插队的一个方法。A、B两个线程，A在运行中调用b.join方法，相当于让B线程插队执行，并且直到B线程执行完，A线程才会接着执行。

dubbo

谈谈你对dubbo的理解

  1. dubbo是一个高性能的轻量级的开源rpc开发框架，它由三层模型构成，第一层呢是Business业务逻辑层，由我们自己来提供接口和实现(service)以及配置信息(config),
    第二层是rpc调用的核心层，负责封装整个rpc的调用过程，负载均衡，集群容错，代理等核心功能，第三个是Romoting，是对网络数据传输协议，和数据转换的封装。

dubbo的工作原理是什么样子

1.服务提供者在启动时，向注册中心注册自己提供的服务。
2.服务消费者在启动时，向注册中心订阅自己所需的服务。
3.注册中心返回服务提供者地址列表给消费者，如果有变更，注册中心将基于长连接推送变更数据给消费者。
4.服务消费者，从提供者地址列表中，基于软负载均衡算法，选一台提供者进行调用，如果调用失败，再选另一台调用。
5.服务消费者和提供者，在内存中累计调用次数和调用时间，定时每分钟发送一次统计数据到监控中心

这是整个dubbo的原理和实现过程

你再说说dubbo的负载均衡算法吧

dubbo有五大负载均衡算法
第一个是加权随机,就是通过区间的随机算法，获取某个目标服务，我们也可以针对某个服务器去增加权重，从而获取更大的被调用可能性。
第二种是最小活跃数，每一个服务一开始的活跃数都为0，每次收到请求，活跃数加一，完成请求后再活跃数减1，性能好的服务处理请求的速度更快，所以活跃数也就越小，
这个时候服务性能更好的服务器就能优先获取请求
第三个是一致性hash，也就是说，根据同一个客户端请求，根据一致性hash算法，会落到一个服务器上，只要请求的信息没有变化的情况下，他的请求永远会落到同一台服务器上，
第四个是加权轮询，也就是按照服务器123123进行轮询，而加权是说我们可以对性能更好的服务器节点去增加权重，从而使得这个服务获得更多的调用。
第五种最短响应时间的权重随机，根据请求影响最快的服务器设置权重，从而得到一个目标服务器节点进行访问。

  3. 数据拆分，分库分表的话很难操作

幻读

幻读是指在你查询的时候，其他事务插入新的数据,导致两次结果不一样
不可重复度读，在你查询的时候，修改或删除了查询的数据，导致两次结果不一样
不可重复读通过行锁和可重复度的隔离级别解决，
在读已提交和可重复度的隔离级别中，有mvvc(多版本并发控制），采用的是版本链，trx id，用于存储对修改的事务id，roll_pointer 记录上一次版本地址，通过undo log，获取上一个版本的信息
解决幻读
1.快照读
只有最简单的select操作是快照读，不加锁,通过mvcc来避免幻读
2.当前读
通过next-key避免幻读，比如update,delete,for update(排他锁,其他线程不能读写），lock in share mode（共享锁,其他线程可以读）,这两个都是悲观锁，乐观锁就是加version字段，
每次修改前，查询当前的version与缓存中是否一致
